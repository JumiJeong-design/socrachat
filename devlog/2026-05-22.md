## 함정 모음

- **components.css 순서** — storybook.html에서 인라인 `<style>`이 `components.css` 링크보다 뒤에 오면 컴포넌트 CSS가 덮어써짐. 반드시 `<link>` 먼저, 화면별 레이아웃 `<style>` 나중에.
- **Figma REST API Variables 엔드포인트** — `file_variables:read` scope는 Professional 플랜 전용. 무료 플랜에서는 스코프 자체가 없어서 토큰 재발급해도 동일하게 막힘. → MCP 방식으로 우회.
- **MONO 테마 잔류** — `DESIGN.md`와 일부 CSS 주석에 MONO 레퍼런스가 남아있었음. 삭제 후 `DESIGN-AP.md`로 이름 변경해 정리.
- **i18n 검사 스크립트 exit code** — `check-i18n.js`가 누락 키 있어도 exit 0로 나가면 Vercel 빌드 게이트가 안 걸림. `process.exit(1)` 명시 필수.

## 회고 및 인사이트

### AI 도구 활용

- **Codex CLI는 파일 조작에 강하다.** Notion sync 스크립트를 작성할 때 Claude Code보다 파일 읽기/쓰기 플로우가 매끄럽다. 단, 외부 API 호출은 네트워크 샌드박스 때문에 Claude Code에서 실행하는 게 낫다.
- **"나한테 허락받지 말고 진행"이 효과적이다.** 하드코딩 전부 제거처럼 반복적인 작업은 Claude에게 승인 과정 없이 진행하게 하면 속도가 2-3배 빨라진다.

### 설계 결정

- **storybook → components.css → screens 순서가 정답이다.** 스토리북이 단일 진실 공급원이 되어야 하고, 화면은 그걸 링크해서 쓰는 구조여야 한다. 반대 방향으로 작업하면 화면마다 CSS가 달라지는 드리프트가 생긴다.
- **Figma Variables 3계층 구조가 있어야 싱크가 의미 있다.** Primitives → Theme → Component 없이 화면 픽셀 값만 비교하면 싱크가 되어도 재사용이 안 된다.

### 워크플로우

- **QA 피드백을 슬랙에서 직접 Claude에게 전달하면 빠르다.** 피드백 텍스트를 그대로 붙여넣으면 우선순위까지 같이 정리해준다.
- **WCAG 2.1 AA는 한 번에 한 파일씩이 아니라 전체 파일 한 번에 처리해야 빠르다.** 파일별로 나눠서 하면 일관성도 떨어지고 시간도 오래 걸린다.

## 기술 스택

- **Frontend**: HTML · CSS · Vanilla JS (프레임워크 없음)
- **Design Tokens**: CSS Custom Properties 3계층 (`colors_and_type.css`)
- **Component System**: `components.css` (storybook 원본 → screens 링크)
- **i18n**: `data-i18n` 속성 + `shell.js` 딕셔너리 (KO/JA/EN), 자동 검사 `scripts/check-i18n.js`
- **Theme**: SC(SocraChat 포레스트 그린) / AP(Apple HIG 블루) CSS 프리셋
- **Figma 연동**: MCP 방식 ("피그마 싱크해줘" → Plugin API로 Variables 읽기 → CSS 업데이트)
- **Figma 디자인 관리**: `HTML Sync - Component Synced KR` 페이지에 HTML/CSS 기반 편집 가능 화면 싱크 완료
- **CI/CD**: GitHub + Vercel (빌드 게이트: i18n 검사 통과 필수)
- **Automation**: Notion sync 스크립트 운영 중 (`scripts/sync-devlog-to-notion.mjs`)
- **AI 도구**: Claude Code (Anthropic), Codex CLI (OpenAI)
