## 함정 모음

- **i18n 딕셔너리 아포스트로피** — 단따옴표 문자열 안에 `'` 포함 시 SyntaxError. 에러 메시지 없이 조용히 전체 딕셔너리가 무력화됨. → 아포스트로피 포함 문자열은 큰따옴표로 감쌀 것
- **TEXTAREA i18n** — `placeholder` 속성이 아니라 `el.textContent`로 바꿔야 반영됨
- **JS `style.setProperty()`는 CSS보다 항상 우선** — tweaks.js처럼 인라인 스타일로 CSS 변수를 주입하면 외부 CSS 파일(`apple-theme.css` 등)이 전부 무시됨. 테마 CSS는 반드시 스코핑(`html[data-design="apple"]`)해서 격리할 것
- **Figma 캡처는 첫 화면만** — `flow.html`처럼 JS로 화면을 동적 전환하는 구조는 첫 화면만 캡처됨. 각 화면 개별 URL로 따로 캡처해야 함

## 회고 및 인사이트

### AI 도구 활용

- **Claude Code + Codex CLI 역할 분리가 자연스럽게 됐다.** Claude Code는 기획·구조 설계·대규모 구현, Codex CLI는 코드 검토·버그 수정 같은 집중적인 작업에 더 잘 맞았다. 두 도구를 같은 프로젝트에서 번갈아 써도 컨텍스트가 유지됐다.
- **"왜 안 되지?"보다 증상을 그대로 보여주는 게 빠르다.** 에러 메시지나 화면 상태를 그대로 붙여넣으면 Claude가 원인까지 파악해줬고, 막연히 물어볼 때보다 해결이 훨씬 빠르다.

### 설계 결정

- **Tweaks 패널 제거가 옳은 선택이었다.** 개발자 관점에서 만들어진 기능이 디자이너 워크플로우에는 오히려 방해가 됐다. "이 기능이 누구를 위한 건가"를 먼저 묻는 게 중요하다.
- **Figma 양방향 자동 연동은 현실적으로 어렵다.** 색상/토큰 레벨은 가능하지만 레이아웃까지는 한계가 있다. "필요한 시점에만 스냅샷" 방식이 유지보수 부담 없이 실용적이다.

### 워크플로우

- **체크리스트를 CLAUDE.md에 박아두는 것이 효과적이다.** 새 세션에서도 Claude가 자동으로 기억하고, `/ship` 한 번으로 i18n 검사 → 커밋 → 배포까지 실행된다. 반복 작업은 명령어로 만들어두는 게 이득이다.
- **작은 디테일(아포스트로피 하나)이 전체를 조용히 망가뜨릴 수 있다.** i18n 딕셔너리 SyntaxError처럼 에러 메시지도 없이 기능 전체가 무력화되는 경우가 있어서, 언어 전환 같은 기능은 3개 언어 전부 실제로 눌러보는 확인이 필수다.

## 플로우 평가 및 다음 단계

### 오늘 플로우 평가

**잘 된 것**

코드 기반으로 디자인 시스템을 구축하는 방향 자체는 맞다. CSS Custom Properties를 토큰으로 쓰고, 스토리북으로 컴포넌트를 문서화하고, SC/AP 테마 전환으로 다른 디자인 언어를 실험한 건 — 피그마 없이도 할 수 있는 것 중에서 탄탄한 기반이다. 하루 만에 7개 화면 + 스토리북 + i18n + 테마 전환까지 나온 건 속도 면에서 분명한 강점이다.

**구조적 한계**

토큰이 1계층(`--accent`, `--bg`)으로 평평하게 설계되어 있다. 실제 서비스 수준의 디자인 시스템은 보통 3계층으로 나뉜다:

```
Primitive    --color-green-600: #2D5F4F
    ↓
Semantic     --color-brand-primary: var(--color-green-600)
    ↓
Component    --button-bg: var(--color-brand-primary)
```

지금은 `--accent` 하나를 버튼 배경, 링크 색상, 포커스 링 등 여러 곳에서 직접 참조하는 구조라, 규모가 커질수록 예상 못한 곳에서 깨진다. 오늘 tweaks.js 충돌이 그 예시였다.

**Figma 연동 방향**

"코드 → Figma 스크린샷" 방식은 피그마를 문서 도구로 쓰는 것이지, 디자인 도구로 쓰는 게 아니다. Home 화면 하나만 컴포넌트 인스턴스로 재조립했고 나머지 6개는 픽셀 캡처본으로 남았다. 방향을 정하지 않으면 앞으로도 관리 부담만 쌓인다.

---

### 디자인 퀄리티를 올리기 위한 추천 프로세스

**1단계 — 토큰 계층 정리 (가장 먼저)**

`colors_and_type.css`를 3계층(Primitive → Semantic → Component)으로 리팩토링한다. 이게 되어야 나머지가 의미 있다. AP 테마 전환처럼 CSS 한 파일만 교체해도 전체가 바뀌는 구조가 만들어진다.

**2단계 — 컴포넌트 접근성 (a11y)**

오늘 만든 Button, Composer, Persona Card에 `aria-label`, `role`, 키보드 포커스 스타일이 없다. 실제 서비스엔 필수다. Claude Code에게 "이 컴포넌트 WCAG 2.1 AA 기준으로 접근성 추가해줘"라고 하면 한 번에 처리 가능하다.

**3단계 — Figma 역할 재정의**

피그마를 완전히 버리기보다 **토큰과 컴포넌트 정의는 코드가 원본, 피그마는 커뮤니케이션 도구**로 역할을 명확히 나누는 게 현실적이다. 오늘 만든 `socrachat-tokens.json`을 Token Studio로 Figma Variables에 연결하면, Figma에서 색상/간격을 바꿨을 때 코드에도 반영되는 흐름이 완성된다.

**4단계 — 컴포넌트 API 설계**

지금은 HTML 레벨에서 컴포넌트를 복붙하는 방식인데, 서비스 수준이 되면 `data-variant="primary"`, `data-size="lg"` 같은 속성으로 컴포넌트를 제어하는 구조가 필요하다. 스토리북에서 variant를 선택하면 컴포넌트 HTML이 바뀌는 형태가 목표다.
